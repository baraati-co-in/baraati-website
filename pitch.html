Proof, not a prototype

A real app, running in production.

Baraati replaces the scattered WhatsApp groups + spreadsheets every Indian wedding runs on today. It's an invite-only React Native / Expo app with a Supabase backend — and it's been through 15+ shipped releases, almost entirely built and iterated on with Claude Code.

🔐

Encrypted govt-ID pipeline

Per-wedding AES-256-GCM envelope encryption for passport/ID collection, with audit-logged access and streamed decrypts — never a raw presigned URL.

🧩

Multi-tenant from day one

Row-level security on every table, hosts and guests scoped per wedding, with a documented recursion trap (NO FORCE RLS) that we hit and fixed for real.

📈

Load-tested, not guessed

A standalone load-simulation framework drives the real backend through the app's own data layer to find wedding-day-scale bottlenecks before guests do.

✨ The actual pitch

A codebase that teaches Claude its own history.

Every project accumulates scar tissue — the recursion bug we fixed once, the migration we won't repeat, the key we can never rotate. Normally that knowledge lives in one engineer's head. We wrote it down as Claude Code skills instead: 16 of them, covering architecture invariants, the security model, a symptom→cause debugging playbook, change-control rules, and a full "failure archaeology" of every dead end we've already walked into — so Claude doesn't propose fixing something that was deliberately built that way.

architecture-contract
Load-bearing invariants, before touching schema/auth/storage.
debugging-playbook
Symptom → likely cause → discriminating check → fix.
failure-archaeology
Every reverted idea and settled debate, so it isn't re-fought.
security-model-reference
RLS, envelope encryption, auth trust model, threat summary.
change-control
What needs approval before it ships — no silent deploys.
scale-campaign
A decision-gated load-test plan for real wedding-day traffic.

In practice

What this actually looks like day to day.

CLAUDE.md

One root file every session reads first — tech stack, repo map, critical systems, known risks. Updated as part of the change, not after.

Skills over prompts

Instead of re-explaining "don't rotate the govt-ID key" every session, it's a loaded skill with the incident behind it — Claude checks before it acts, not after it breaks something.

Multi-agent review

A dedicated review skill runs multi-agent verification against the diff before anything ships — correctness bugs and simplification passes, not just "looks fine to me."

Persistent memory

A separate auto-memory system tracks project state release-over-release, so decisions made months ago (why RLS is NO FORCE, why a key can't rotate) aren't relitigated.
